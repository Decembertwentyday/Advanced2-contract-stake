# MetaNode Stake 前端（stake-fe）架构与代码导读

> 文件路径：`stake-fe/docs/前端架构与代码导读.md`  
> 各源码文件内另有 **中文块注释**，与本篇互为补充（ABI 大文件仅在文件头说明）。

本文面向刚学完 wagmi / ethers / viem 的开发者，说明本仓库 **如何组织页面、如何连钱包、如何读链与发交易**。按文中顺序阅读源码，可以建立完整心智模型；文末简述如何在此基础上改用 **ethers** 或 **wagmi + ethers**。

---

## 1. 技术栈各自做什么

| 技术 | 在本项目中的角色 |
|------|------------------|
| **Next.js（Pages Router）** | 路由入口在 `src/pages/`：每个文件/文件夹对应一个 URL，`_app.tsx` 包裹所有页面。 |
| **React** | 组件 + 状态（`useState`）+ 副作用（`useEffect`）。 |
| **wagmi** | React 里用钩子拿「当前链、账户、WalletClient」；和 RainbowKit 一起完成连接钱包。 |
| **viem** | 底层以太坊工具库：编码调用数据、`parseUnits`、`waitForTransactionReceipt`、通过 `getContract` 得到可 `read`/`write` 的合约实例。 |
| **RainbowKit** | 连接钱包 UI（`ConnectButton`）+ 依赖 WalletConnect 的 `projectId`。 |
| **TanStack Query** | wagmi v2 用它来缓存链上请求；必须在 `WagmiProvider` 里包一层 `QueryClientProvider`。 |
| **MUI ThemeProvider** | 仅为主题色/字体；大部分 UI 用 Tailwind。 |

核心思想：**wagmi 管「钱包与 React 状态」**，**viem 管「和链对话的协议细节」**；本项目的合约实例是 **viem 的 `getContract`**，再交给页面里的 `write` / `read`。

## 2. 建议阅读顺序（从根到叶）

按下面顺序打开文件，你就能从「应用从哪启动」跟到「质押按钮点了发生什么」。

1. **`src/pages/_app.tsx`**  
   全局壳：注入 Wagmi、React Query、RainbowKit、Toast、布局。所有页面共享这里的 Provider。

2. **`src/pages/index.tsx`**  
   站点根路径 `/` 对应的页面组件。这里非常薄：直接渲染 `home/page` 里的首页。

3. **`src/components/Layout.tsx`**  
   每一页外面的统一框架：背景、顶栏 `Header`、主内容区、`children` 就是当前路由页面。

4. **`src/components/Header.tsx`**  
   导航链接（Stake / Withdraw / Claim）+ `ConnectButton`。

5. **`src/pages/home/page.tsx`**  
   首页业务：质押（ETH 或 ERC20）、以及本页内嵌的领取奖励按钮。

6. **`src/pages/withdraw/index.tsx`**  
   解质押 `unstake` + 到期后 `withdraw`。

7. **`src/pages/claim/index.tsx`**  
   独立领取页：`claim`。

8. **`src/hooks/useContract.ts`**  
   把「当前链 ID + 钱包 WalletClient」合成一个 **viem 合约实例**。

9. **`src/utils/contractHelper.ts`**  
   关键：**`public` 客户端**（HTTP RPC 读链、模拟）+ **`wallet` 客户端**（签名发交易）。

10. **`src/utils/wagmi.ts`** + **`src/utils/sepoliaTransport.ts`** + **`src/utils/viem.ts`**  
    链配置、RPC 传输（含 fallback）、给 `contractHelper` 用的 `PublicClient`。

11. **`src/hooks/useRewards.ts`**  
    聚合读合约：`pool`、`user`、`stakingBalance` 等，供多个页面复用。

12. **`src/utils/env.ts`**、**`src/utils/index.ts`**  
    合约地址、`Pid`（池子 ID，本项目为 `0`）。

读完这一遍，你应能回答：**数据从哪来（read）**、**交易从哪发（write）**、**为什么要有两个 client（public + wallet）**。

## 3. Next.js 路由与本项目 URL

使用 **Pages Router**（不是 `app/` 目录）：

| 文件路径 | 浏览器路径 |
|----------|------------|
| `pages/index.tsx` | `/` |
| `pages/withdraw/index.tsx` | `/withdraw` |
| `pages/claim/index.tsx` | `/claim` |
| `pages/_app.tsx` | 不对应 URL，是全局包装 |

`pages/home/page.tsx` **不是**独立路由，只是被 `index.tsx` import 的一个组件（放在 `pages/home/` 仅为文件组织习惯）。

## 4. `_app.tsx`：Provider 嵌套顺序（由内到外理解）

渲染树（简化）：

```
ThemeProvider (MUI)
  └ WagmiProvider (wagmi 全局配置 config)
      └ QueryClientProvider (React Query，wagmi 依赖)
          └ RainbowKitProvider (连接钱包 UI)
              └ ToastContainer
              └ Layout
                    └ Component (当前页面：index / withdraw / claim)
```

**为什么要这个顺序？**

- `WagmiProvider` 必须包住所有使用 `useAccount`、`useWalletClient` 的组件。
- wagmi v2 要求 **`QueryClientProvider` 在 `WagmiProvider` 内部**（或按官方文档对齐），否则钩子无法正常工作。
- `RainbowKitProvider` 依赖 wagmi 的上下文，所以要放在 `WagmiProvider` 里。

## 5. 合约交互：read 与 write 的分工

### 5.1 两个 Client

在 `contractHelper.ts` 里，viem 的 `getContract` 接收：

- **`public`**：`createPublicClient` + HTTP RPC（`viem.ts` / `sepoliaTransport`）。  
  - 用于 **`read`**：查余额、查池子、查用户奖励。  
  - 用于 **`write` 前的模拟 / 估算**（viem 内部会走只读 RPC），所以 RPC 不稳定时，质押前就会报错。

- **`wallet`**：来自 wagmi 的 `useWalletClient()`，本质是用户钱包（如 MetaMask）提供的 signer。  
  - 用于 **`write`**：真正发起交易、由用户签名。

### 5.2 一笔质押在代码里分几步

以 ETH 池为例（`home/page.tsx`）：

1. `stakeContract.write.depositETH(..., { value: amountWei })`  
   - 通过 **WalletClient** 发交易，返回 **tx hash**。
2. `waitForTransactionReceipt(walletClient, { hash })`  
   - 等待链上确认；用钱包 client 或兼容的 client 轮询收据。

ERC20 池多两步：**先 `approve` 质押合约**，再 `deposit`，每步都可等待 receipt。

### 5.3 为什么用 `Pid`

合约支持多个池子（pool），前端常量 **`Pid = 0`** 表示始终操作 **0 号池**。换池子只需改常量或改成 UI 选择。

## 6. 环境变量

| 变量 | 作用 |
|------|------|
| `NEXT_PUBLIC_STAKE_ADDRESS` | 质押合约地址；未设置时 `env.ts` 会落到 `zeroAddress`，容易导致调用失败，部署后务必配置。 |
| `NEXT_PUBLIC_INFURA_API_KEY` | 可选；用于拼接 Infura Sepolia URL，见 `sepoliaTransport.ts`。 |

以 `NEXT_PUBLIC_` 开头会被 Next 打进浏览器端代码，**不要放私钥**。

## 7. 自定义 Hooks 与工具

- **`useStakeContract`**：封装质押合约 ABI + 地址。  
- **`useTokenContract`**：按池子的 `stTokenAddress` 动态创建 ERC20 实例（`approve` / `decimals`）。  
- **`useRewards`**：定时/手动刷新用户与池子数据；内部用 `retryWithDelay` 缓解 RPC 限流。  
- **`retry.ts`**：简单重试 + 对 429 做指数退避。  
- **`metamask.ts`**：`wallet_watchAsset`，把 MetaNode 代币加到钱包（可选功能）。

## 8. UI 组件

- **`components/ui/Button.tsx`**、`Card.tsx`、`Input.tsx`：样式封装，与链无关。  
- **`Layout` / `Header`**：壳与导航。

## 9. 常见排查思路

- **交易还没弹出钱包就失败**：多半是 **public RPC** 模拟/估算失败；已用 `fallback` 多 RPC，仍失败时换 Alchemy 等稳定节点。  
- **合约 revert**：链上逻辑拒绝（额度、池子类型、权限等），需在合约或 Etherscan 上看 revert 原因。  
- **地址为零**：检查 `NEXT_PUBLIC_STAKE_ADDRESS`。

## 10. 你计划做的改造：ethers 与 wagmi + ethers

### 10.1 纯 ethers（v6）思路

- 用 `BrowserProvider(window.ethereum)` 得到 `Signer`，用 `Contract(address, abi, signer)`。  
- `read` 可用 `provider.readContract` 或 `contract.foo.staticCall`。  
- 不再使用 `viem.getContract`，但 **ABI 与地址** 仍可与现在共用。  
- 需自己处理：链切换、账户变化、事件刷新；或继续用 wagmi 只拿 `address`/`chainId`，合约层全用 ethers。

### 10.2 wagmi + ethers

- wagmi 提供 **`useEthersSigner`**（来自 `@wagmi/ethers` 等适配包，以官方当前文档为准）把 wagmi 的 wallet client 转成 ethers `Signer`。  
- 好处：保留 RainbowKit 连接体验，合约调用写法用你熟悉的 ethers。

### 10.3 迁移时建议保留的边界

- **页面与 Hooks 分层**：保持「UI 只调 `handleStake` / `handleClaim`」，底层换成 ethers `Contract` 即可。  
- **常量**：`Pid`、`StakeContractAddress`、链 ID 仍集中在一处。  
- **等待确认**：ethers v6 使用 `tx.wait()` 替代 viem 的 `waitForTransactionReceipt`（参数与返回值不同，需对照文档）。
